## 1. Безопасность

### 1.1 [CRITICAL] Обход платного доступа к видео (Broken Access Control / IDOR)
**`src/app/api/videos/route.ts:9-34`**

Эндпоинт стриминга видео проверяет только факт аутентификации (`getUser()`), но **не
проверяет доступ к конкретному уроку**. Вся логика доступа (публичность урока, подписка
«Все включено», индивидуальный `lessonAccess`, срок действия) реализована отдельно в
`getLesson()` (`src/app/lib/dal/lesson.dal.ts:10-58`), которая при отсутствии доступа
возвращает `videoURL: ""`. Но видео отдаётся из независимого роута, принимающего
произвольный `?name=<файл>`. Имена файлов — это `path.basename` исходной загрузки
(предсказуемые). 

**Сценарий атаки:** любой залогиненный пользователь (в т.ч. бесплатный «Ознакомительная»)
запрашивает `/api/videos?name=lesson1.mp4` и скачивает **любое** премиум-видео, полностью
минуя paywall. Санитизация пути (`path.resolve` + `startsWith`) защищает от traversal, но не
от несанкционированного доступа к легитимным файлам.

→ **Исправление:** не принимать сырое имя файла от клиента. Принимать `lessonId`, находить
`videoURL` в БД и прогонять через ту же проверку доступа, что и `getLesson()`
(статус/подписка/`lessonAccess`/срок) до отдачи потока. Опционально — signed-URL, привязанные
к пользователю и уроку.

```ts
// было: filename берётся напрямую из query
const filename = req.nextUrl.searchParams.get("name");

// стало (набросок):
const lessonId = Number(req.nextUrl.searchParams.get("lessonId"));
const lesson = await getLesson(lessonId);           // включает проверку доступа
if (!lesson || "forbidden" in lesson || !lesson.videoURL) {
  return NextResponse.json({ error: "Forbidden" }, { status: 403 });
}
const filepath = getVideoPath(lesson.videoURL);      // имя из БД, не от клиента
```

### 1.2 [HIGH] Отсутствие валидации `SESSION_SECRET` → возможна подделка JWT
**`src/app/lib/session.ts:6-7`**

```ts
const secretKey = process.env.SESSION_SECRET;
const encodedKey = new TextEncoder().encode(secretKey); // при undefined — пустой ключ
```

Если переменная окружения не задана, ключ пустой: токены подписываются и проверяются пустым
секретом и **тривиально подделываются**. Злоумышленник кует JWT с `role: "admin"` и получает
полный административный доступ. Нет fail-fast и проверки минимальной длины.

→ **Исправление:** на старте бросать ошибку, если `SESSION_SECRET` не задан или короче 32 байт.

```ts
const secretKey = process.env.SESSION_SECRET;
if (!secretKey || secretKey.length < 32) {
  throw new Error("SESSION_SECRET must be set and at least 32 chars");
}
const encodedKey = new TextEncoder().encode(secretKey);
```

### 1.3 [HIGH] Логин совмещён с авторегистрацией, нет rate-limiting
**`src/app/actions/auth.ts:12-116`**

`signin` при отсутствии email **молча создаёт нового пользователя** и логинит его. Нет
отдельной регистрации, подтверждения email, rate-limiting, CAPTCHA. Последствия:
- неограниченное создание аккаунтов (спам / исчерпание ресурсов, «мусорные» подписки);
- account enumeration: существующий email + неверный пароль → ошибка; несуществующий →
  редирект (различимое поведение);
- `bcrypt.hash(password, 10)` вычисляется на **каждой** попытке ещё до поиска пользователя
  (лишняя нагрузка, мягкий DoS-вектор).

→ **Исправление:** разделить вход и регистрацию; добавить rate-limiting (по IP/email); не
вычислять хеш на ветке существующего пользователя.

### 1.4 [HIGH] Логирование и возврат пароля в открытом виде
**`src/app/actions/auth.ts:21, 25-27, 44-47, 73-79`** и **`src/app/components/signup-form/SignupForm.tsx:35`**

При ошибке валидации `console.log(validatedFields)` пишет **пароль в открытом виде в логи
сервера**. Кроме того, action возвращает пароль обратно в `fields`, а форма подставляет его
в `defaultValue={state?.fields.password}`. Пароли в логах = утечка учётных данных.

→ **Исправление:** никогда не логировать объект с паролем; не возвращать пароль клиенту.

### 1.5 [MEDIUM] Раскрытие внутренней информации в ответе об ошибке
**`src/app/api/videos/route.ts:109-113`**

`details: JSON.stringify(error)` отдаёт клиенту сырую ошибку (пути ФС, стек).

→ **Исправление:** возвращать обобщённое сообщение, детали — только в серверный лог.

### 1.6 [MEDIUM] Эндпоинты прогресса без проверки доступа к уроку
**`src/app/api/lessons/[lessonId]/progress/route.ts:11-98`** + **`lesson.dal.ts:113-226`**

GET/POST проверяют только `getUser()`, но не доступ к уроку. Любой пользователь может
читать/писать прогресс произвольного `lessonId` (загрязнение данных, косвенная проверка
существования уроков через аналитику).

→ **Исправление:** проверять доступ к уроку перед чтением/записью прогресса (переиспользовать
логику `getLesson`).

### 1.7 [MEDIUM] Слабая парольная политика
**`src/app/lib/definitions.ts:5-13`**

Минимум 5 символов, проверки сложности закомментированы.

→ **Исправление:** поднять минимум до 8–12 символов, добавить проверку сложности/словарь
(например, zxcvbn).

### 1.8 [LOW / требует проверки] CSRF на мутационных API-роутах
**`src/app/api/lessons/lesson/route.ts`, `.../progress/route.ts`, `/api/videos`**

Server Actions в Next 15 имеют встроенную проверку Origin, а cookie сессии — `sameSite:"lax"`,
что для POST/PATCH/DELETE в основном блокирует cross-site. Явной проверки Origin/CSRF-токена
на кастомных роутах нет. Риск невысокий, но **требует проверки** — рекомендуется явная
валидация заголовка `Origin` на изменяющих роутах.

### 1.9 [ИНФО] Проверено и НЕ является проблемой
- **SQL-инъекции:** не найдено. Все запросы — параметризованный билдер Drizzle; шаблоны
  `sql`` `` содержат только ссылки на колонки. Конкатенации пользовательского ввода в SQL нет.
- **XSS:** React экранирует по умолчанию; `dangerouslySetInnerHTML`/`innerHTML` не
  используются; `program`/`description` рендерятся как текст. Инлайн-скрипт Yandex.Metrika
  использует `yid` из env (не пользовательский ввод).
- **Command injection:** `child_process`/`exec`/`eval`/`new Function` не используются.
- **Path traversal:** в `/api/videos` и `getVideoPath` применён `path.basename` +
  `resolve` + проверка `startsWith(videosDir + path.sep)` — корректно.
- **Хранение секретов:** `.env*` в `.gitignore`; секретов в коде/истории git не найдено.
- **Зависимости/CVE:** [требует проверки] запустить `npm audit` (в аудите нет доступа к сети).
  Ключевые версии свежие: `next@15.4.8`, `jose@6`, `bcrypt@6`, `drizzle-orm@0.44`.

## 2. Архитектура и надёжность

### 2.1 [HIGH] Нетранзакционные многошаговые записи
**`src/app/actions/courses.ts:65-95` (create), `:197-237` (update); `src/app/actions/modules.ts:152-174`**

`createCourse`/`updateCourse`/`updateModule` делают вставку/удаление/вставку связей
отдельными `await` без транзакции. При падении на любом шаге БД остаётся в несогласованном
состоянии (курс без модулей; потерянные навыки после удаления старых связей и падения вставки
новых). Транзакцию использует только регистрация в `auth.ts:83-104`.

→ **Исправление:** обернуть связанные записи в `db.transaction(async (tx) => { ... })`.

### 2.2 [MEDIUM] `verifySession` падает при удалённом пользователе
**`src/app/lib/dal.ts:22-27`**

```ts
const getSessionID = await db.select(...).limit(1);
if (getSessionID[0].session !== session.sessionID) { ... } // getSessionID[0] может быть undefined
```

Если пользователь удалён при живой сессии, `getSessionID[0]` — `undefined` → TypeError →
необработанная 500 на каждом запросе.

→ **Исправление:** `if (!getSessionID[0]) redirect("/api/auth/clear-session")` перед доступом
к `.session`.

### 2.3 [MEDIUM] Race condition в upsert прогресса
**`src/app/lib/dal/lesson.dal.ts:132-179` (`updateLessonProgress`), `:181-226` (`completeLessonProgress`)**

Обе функции делают check-then-insert без транзакции/`onConflict`. Плеер шлёт `saveProgress`
одновременно на `timeupdate`, `pause`, `beforeunload` (`Player.tsx:87-121`), а POST-роут
вызывает `updateLessonProgress`, затем `completeLessonProgress`. Два параллельных insert по
одному PK `(userId, lessonId)` → нарушение уникальности → 500.

→ **Исправление:** заменить на `insert(...).onConflictDoUpdate(...)` (атомарный upsert).

### 2.4 [MEDIUM] Race condition в разборе MP4 (общий буфер) + невалидная длительность
**`src/app/api/lessons/lesson/route.ts:48-49, 109-117`**

`buff`/`header` объявлены на уровне модуля и переиспользуются между **параллельными**
загрузками → порча данных. Плюс: `start = buffer.indexOf(header) + 16`; если `mvhd` нет в
первых 100 байтах (частый случай — атом `moov` в конце файла), `indexOf` вернёт `-1`,
`timeScale` может быть 0 → `duration = NaN` → колонка `duration integer NOT NULL` → падение
вставки или мусорные данные.

→ **Исправление:** буфер создавать на запрос; парсить mp4 надёжно (или через ffprobe);
проверять `timeScale > 0`.

### 2.5 [LOW] Проглатывание ошибок в DAL
**`course.dal.ts`, `lesson.dal.ts` (повсеместно `catch → return []/null`)**

Ошибки БД маскируются: `getAllCourses` при сбое вернёт `[]`, и UI покажет «нет курсов»
вместо ошибки. Затрудняет диагностику инцидентов.

→ **Исправление:** различать «пусто» и «ошибка»; пробрасывать/логировать с контекстом.

### 2.6 [LOW] Повторный `getUser()` внутри catch-блоков API-роутов
**`src/app/api/lessons/lesson/route.ts:150, 297, 384`**

В обработчике ошибки снова вызывается `getUser()` (доп. запрос к БД; `verifySession` может
инициировать `redirect()`), что усложняет путь ошибки.

→ **Исправление:** получать пользователя один раз в начале и переиспользовать.

## 3. Качество кода

### 3.1 [MEDIUM] Отсутствуют тесты
Нет ни одного теста и скрипта `test` в `package.json`. Критические пути (аутентификация,
контроль доступа, прогресс, транзакции) не покрыты — высокий риск регрессий для платной
платформы.

→ **Исправление:** добавить тесты на auth/access-control/progress (Vitest + тестовая БД).

### 3.2 [MEDIUM] Дублирование boilerplate аудита
**`courses.ts`, `modules.ts`, `lessons.ts`, `api/lessons/lesson/route.ts`** — блок
try/catch + `logAdminAction` (success/failure) скопирован ~10 раз (нарушение DRY).

→ **Исправление:** вынести обёртку `withAudit(action, meta)`.

### 3.3 [LOW] Дублирование «плоского списка уроков»
**`course.dal.ts:397-404` и `:429-436`, `[lessonId]/page.tsx:48-54`, `lessons/page.tsx:59-65`** —
одна и та же логика сплющивания модулей→уроков в 4 местах.

→ **Исправление:** вынести `getFlatLessonList(course)`.

### 3.4 [LOW] Мёртвый код
`src/app/utils/db.ts` (полностью закомментирован), `src/app/lib/dto.ts` (закомментирован),
seed-блок в `src/db/db.ts:9-29`, закомментированные `materials`/`Skill` в страницах уроков.

→ **Исправление:** удалить.

### 3.5 [LOW] Мелочи типобезопасности/консистентности
`getUser` использует `findMany` + `[0]` вместо `findFirst` (`dal.ts:40-49`); множественные
касты `currentUser.role as "admin"` в вызовах аудита (для manager — семантически неверный
кастинг, значение верное); `Readable.fromWeb(stream as any)`.

→ **Исправление:** заменить на `findFirst`; убрать лишние касты.

## 4. Производительность

### 4.1 [MEDIUM] Повторные полные загрузки дерева курса
**`[lessonId]/page.tsx:42` + `lessons/layout.tsx:23-27` + server actions `onNext/onPrevious` (`:65-83`)**

На одну навигацию по уроку дерево курса (`getCourseById`, вложенные модули→уроки) грузится
несколько раз: страница урока + layout + `getCourseProgress` + `getCompletedLessonIds` +
переходы next/previous (снова `getCourseById`). При этом `getCourseProgress` и
`getCompletedLessonIds` выполняют почти идентичный запрос «уроки курса + завершённые».

→ **Исправление:** обернуть `getCourseById` в React `cache()` для дедупликации; объединить
прогресс/завершённые в один запрос; для соседних уроков — лёгкий запрос вместо полного дерева.

### 4.2 [MEDIUM / требует проверки] Отсутствует индекс на часто запрашиваемом FK
**`src/db/schema/users.ts:26` (`subscription.userId`)**

`subscription.userId` — FK без индекса, а выборка по нему частая (`getUserSubscription`,
`updateSubscription`). Связующие таблицы имеют составной PK `(a,b)`, что покрывает поиск по
ведущей колонке, но не по второй; критичные запросы в основном используют ведущую колонку.

→ **Исправление:** добавить индекс на `subscription.userId`; проверить планы запросов по
фактическим данным.

### 4.3 [LOW] `Cache-Control: no-store` на всех видео-ответах
**`src/app/api/videos/route.ts:75, 105`**

Полностью отключает кэширование крупных видео на клиенте/CDN → рост трафика и задержек.

→ **Исправление:** подобрать политику (например, `private, max-age`), совместимую с моделью
доступа.

### 4.4 [LOW] Загрузка видео на локальную ФС в процессе запроса
**`api/lessons/lesson/route.ts` + `utils/helpers.ts:6-10`**

Файлы пишутся в `src/videos` (в дереве исходников), поток буферизуется на диск в рамках
запроса; при 1000 МБ это надолго занимает воркер и несовместимо с serverless.

→ **Исправление:** выносить в объектное хранилище (S3-совместимое) с загрузкой напрямую /
по signed-URL.
